# Evaluation — Georeferencing Quality Metrics

This folder contains all scripts and notebooks for evaluating the quality of LiDAR georeferencing in GNSS outage scenarios. It compares four reconstruction methods:

| Method | Description |
|---|---|
| **INS-only** | Pure inertial navigation (no GNSS, no LiDAR constraints) |
| **F2B** | Front-to-Back LiMatch constraints within each scan pass |
| **S2S** | Scan-to-Scan LiMatch constraints across scan passes |
| **Combined** | F2B + S2S constraints combined |

---

## georef_eval — RMSE Computation

The RMSE evaluation compares a **target** georeferenced cloud (from INS-only / F2B / S2S / Combined) against a **reference** cloud (from ODyN with full GNSS, treated as ground truth). Matching is done by GPS timestamp: each point is uniquely identified by its acquisition time, so no spatial registration is needed.

### Workflow

```
1. python generate_rmse_configs.py --config gen_config_outage_1.yml
       → one per-scan YAML + one run_rmse_*.sh per method

2. bash <output_dir>/<method>/run_rmse_<outage_id>_<method>.sh
       → calls rmse_streaming.py for each scan, writes *.summary.csv

3. analyse_rmse.ipynb  →  aggregate CSVs, plots, comparison tables
```

---

### Step 1 — `generate_rmse_configs.py`

Generates one YAML + one shell script per method. The matching strategy finds the ODyN reference scan with the largest temporal overlap with each target scan.

```bash
python generate_rmse_configs.py --config gen_config_outage_1.yml
```

#### Generator config format

```yaml
outage_id: "outage_1"

ref_dir:      "/path/to/ODyN/merged/ALL"
ref_manifest: "/path/to/ODyN/merged/ALL/merged_manifest.csv"

output_dir: "/path/to/rmse_output"

time:
  buffer: 2.0       # Extra seconds added around each scan window

las:
  scale: 0.001      # LAS output scale

decimate: 10         # Keep 1 point every N in target cloud (memory saving)

methods:
  - name: "INS_only"
    dir:      "/path/to/INS_only/merged/ALL"
    manifest: "/path/to/INS_only/merged/ALL/merged_manifest.csv"

  - name: "F2B"
    dir:      "/path/to/F2B/merged/ALL"
    manifest: "/path/to/F2B/merged/ALL/merged_manifest.csv"

  - name: "S2S"
    dir:      "/path/to/S2S/merged/ALL"
    manifest: "/path/to/S2S/merged/ALL/merged_manifest.csv"

  - name: "Combined"
    dir:      "/path/to/Combined/merged/ALL"
    manifest: "/path/to/Combined/merged/ALL/merged_manifest.csv"
```

**Output:** for each method, creates `<output_dir>/<method_name>/`:
- One `rmse_<outage_id>_<method>_scanNNN.yml` per scan
- One `run_rmse_<outage_id>_<method>.sh` shell script

---

### Step 2 — Run the generated shell script

```bash
bash <output_dir>/F2B/run_rmse_outage_1_F2B.sh
```

The `.sh` is generated by `generate_rmse_configs.py` and calls `rmse_streaming.py` sequentially for each scan — **you do not run `rmse_streaming.py` directly**. Each call processes one (reference, target) scan pair and writes a `.summary.csv` next to its YAML config.

For reference, the per-scan YAML config format used internally:

```yaml
reference:
  path: "/path/to/ref_scan.las"

targets:
  - path:    "/path/to/target_scan.las"
    outfile: "/path/to/output_error.las"   # LAS with e3d extra dim

time:
  outage_start:    305120.0   # GPS time of outage start
  outage_duration: 580.0      # Duration (already includes buffer)
  buffer:          0.0        # Additional buffer (added to outage_duration)

io:
  delim: ","          # Delimiter for TXT files (ignored for LAS)

las:
  scale: 0.001        # Scale for output LAS

decimate: 1           # Decimate target cloud (1 = no decimation)
```

#### Metrics computed

| Metric | Description |
|---|---|
| `N` | Number of matched points |
| `RMSE` | Root mean square 3D error |
| `mean`, `median` | Mean / median 3D error |
| `p95`, `p99` | 95th / 99th percentile |
| `max` | Maximum 3D error |
| `%>0.1m`, `%>0.5m`, `%>1.0m` | Fraction of points exceeding threshold |
| `dx_mean/std`, `dy_mean/std`, `dz_mean/std` | Per-axis bias and dispersion |

Results are saved to a `.summary.csv` next to the config file. The output LAS stores the `e3d` (3D error in metres) as an extra dimension for visualisation in CloudCompare.

#### Supported input formats

| Format | Notes |
|---|---|
| `.las` / `.laz` | GPS time from `gps_time` field. Chunked reading (2M pts). |
| `.txt` / `.csv` | First column must be GPS time. Comma-delimited. |

---

### Step 3 — `analyse_rmse.ipynb`

Loads all `*.summary.csv` files, aggregates metrics by study zone / scan, and produces:
- Summary tables comparing INS-only / F2B / S2S / Combined
- RMSE time series along the outage window
- Per-zone box plots

Open with Jupyter:

```bash
jupyter notebook Evaluation/georef_eval/analyse_rmse.ipynb
```

## L2L_eval — L2L Constraint Residuals

**`L2L_eval/L2L_S2S.ipynb`**

Analyses the LiDAR-to-LiDAR constraint residuals produced by the S2S combined trajectory. This is used to assess the quality and consistency of the S2S LiMatch correspondences before and after ODyN optimisation.

> Note: This notebook is also used as an intermediate step to compute the APX15 Combined trajectory.

---

## LCP_eval — LCP-Based Evaluation

**`LCP_eval/LCP_ECCR.ipynb`**

Evaluates georeferencing quality using LCP (Linear Control Points) or check-point–based metrics on the ECCR dataset. Provides an independent quality assessment that does not rely on the ODyN reference trajectory.
